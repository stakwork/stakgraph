# Phase 3 — Swarm Handoff: Provisioning Bifrost Admin Credentials

> Concrete plan for how a fresh stakgraph-gateway container deployed
> by sphinx-swarm gets its Bifrost admin credentials into Hive's DB
> so Hive's phase-1 reconciler can call `/api/governance/*` AND so
> Hive's frontend can display "visit your LLM dashboard" with a
> user-copyable password.
>
> Companion to `phase-1-reconciler.md` (which assumes Hive already has
> the admin user + password) and the gateway README's "Authentication"
> section (which describes how the gateway image's auth is enforced).

## Trust boundary at a glance

Two credentials are in play. They serve different audiences and are
exposed differently:

| Credential | Generated by | Stored where | Server-only? | Shown to user? | Used for |
|---|---|---|---|---|---|
| `stakwork_secret` (a.k.a. `BIFROST_PROVISIONING_TOKEN`) | swarm (pre-existing; same value used by boltwall, repo2graph, etc.) | swarm super-admin state + Hive backend env/config | **YES — never leaves Node.js process** | NO | One-time bootstrap: Hive backend → `/_plugin/admin-credentials` |
| `bifrost_admin_password` | swarm (newly generated per gateway, 16 chars) | swarm super-admin state, Hive DB (encrypted), gateway container env | NO — must be decryptable for UI display | YES (workspace admins) | (a) Hive backend → Bifrost `/api/governance/*`; (b) human admin → Bifrost dashboard login |

**Critical**: the `stakwork_secret` MUST NOT be sent to the frontend.
It's the bootstrap key that proves Hive's backend is talking to swarm
(and grants permission to read the admin password out of any gateway
in the swarm fleet). If it leaks to the browser, anyone with browser
access can call `/_plugin/admin-credentials` on any swarm-deployed
gateway and own every Bifrost in the system.

The `bifrost_admin_password` is meant to be user-facing — that's the
"copy this to log in to your dashboard" experience. It's per-gateway
and only grants control over one workspace's Bifrost.

## Status

The gateway image **already** ships with everything it needs for this
to work:

- `data/config.json` declares `auth_config` referencing
  `env.BIFROST_ADMIN_USER` / `env.BIFROST_ADMIN_PASS`. Bifrost
  resolves these on first boot, bcrypt-hashes the password, and
  persists the hash in `config.db`. The `env.*` reference is kept
  alongside the hash so a rebuild with a different value picks up
  the new password automatically (verified by Bifrost's own test
  `password_change_flushes_existing_sessions`).
- The plugin (`main.go`) runs an in-process HTTP server on
  `127.0.0.1:8189` exposing `GET /_plugin/admin-credentials`
  (Bearer auth, token = `$BIFROST_PROVISIONING_TOKEN`). The
  endpoint returns `{"admin_username":"…","admin_password":"…"}`.
- The wrapper binary (`wrapper/main.go`) fronts both bifrost-http
  (loopback:8181) and the plugin HTTP server (loopback:8189) on a
  single public port, so `/_plugin/*` is reachable via the same
  TLS-terminated host:port as the dashboard.
- The gateway repo's smoke tests confirm all of the above end-to-end
  including password rotation (a rebuild with a different
  `BIFROST_ADMIN_PASS` cleanly invalidates the old password and
  starts accepting the new one).

What's **not yet done** is the swarm-side wiring and the hive-side
consumption. This document specifies both.

Two credentials, two trust levels:

- **`stakwork_secret`** is pre-existing — it already flows from swarm
  → boltwall → repo2graph (`sphinx-swarm/src/images/repo2graph.rs:97`)
  and is held by Hive's server config. We pass the same value to the
  gateway as `BIFROST_PROVISIONING_TOKEN`. Hive's backend presents it
  as a Bearer token to `/_plugin/admin-credentials`; the plugin
  verifies in constant time and returns the admin pair. **Server-side
  only** — never reaches the browser.

- **`bifrost_admin_password`** is generated per-gateway (16 chars,
  see B1.2 below). Hive's backend uses it for `/api/governance/*`
  calls AND surfaces it to authenticated workspace admins via a
  separate dashboard endpoint so the UI can render "visit your LLM
  dashboard" with copy-to-clipboard.

**Why a separate provisioning token instead of just reusing the
admin password directly?** Hive cannot know the admin password until
it has been generated, which happens inside the swarm-provisioned
container. The stakwork secret IS pre-known to both sides. So:
- `stakwork_secret`: pre-shared, used **once** by Hive to bootstrap
  itself with the admin password.
- `admin_password`: generated per-gateway, retrieved by Hive, then
  used for all subsequent Bifrost admin API calls AND shown to the
  workspace admin in Hive's UI.

## Step B1 — sphinx-swarm changes

File: `sphinx-swarm/src/images/bifrost.rs`.

Today the image points at `maximhq/bifrost:latest` (statically linked,
cannot load Go plugins) and only forwards provider API keys. The
changes are:

### B1.1 Switch the image

```rust

            repo: "stakgraph-gateway".to_string(),  // ← change
            root_volume: "/app/data".to_string(),
        }
    }
}
```

(Whatever org/repo we publish to. The image is built from this repo's
`gateway/Dockerfile`.)

### B1.2 Add admin credential fields

Mirror `Neo4jImage`'s pattern (see `sphinx-swarm/src/images/neo4j.rs:48`
where `password: secrets::random_word(32)` is generated at struct
construction). Use **16 characters** — that's the password a workspace
admin will copy from Hive's UI to log into Bifrost's dashboard, and
32 chars is unwieldy for that flow. 16 chars from the `Alphanumeric`
alphabet (62 symbols) yields ~95 bits of entropy, far above what
bcrypt protects:

```rust
#[derive(Serialize, Deserialize, Debug, Clone, Eq, PartialEq)]
pub struct BifrostImage {
    pub name: String,
    pub version: String,
    pub port: String,
    pub host: Option<String>,
    pub links: Links,
    pub admin_user: String,           // ← new, hardcode "admin"
    pub admin_password: String,       // ← new, secrets::random_word(16)
}

impl BifrostImage {
    pub fn new(name: &str, version: &str) -> Self {
        Self {
            // …existing fields…
            admin_user: "admin".to_string(),
            admin_password: secrets::random_word(16),
        }
    }
}
```

Because `BifrostImage` is `Serialize` + `Deserialize`, swarm's existing
state persistence layer automatically saves the generated password
across restarts — no extra code needed.

**Note on the alphabet.** `secrets::random_word` uses rand's
`Alphanumeric` distribution: `[A-Za-z0-9]`. That alphabet contains
the visually ambiguous pairs `O`/`0` and `1`/`l`/`I`. For a
copy-paste user flow this almost never matters, but a workspace
admin reading the password aloud could trip on it. Optional follow-up:
add a `secrets::random_user_friendly_word(n)` to swarm that drops
those characters, and use it here. Not blocking — keep
`random_word(16)` for v1.

### B1.3 Inject env vars into the container

In `pub fn bifrost(img: &BifrostImage) -> Config<String>`, accept the
boltwall image as an optional argument (so we can read its
`stakwork_secret`) — mirror what `repo2graph.rs` does — and extend
`env`:

```rust
pub fn bifrost(img: &BifrostImage, boltwall: Option<&BoltwallImage>) -> Config<String> {
    // …existing env vec…

    env.push(format!("BIFROST_ADMIN_USER={}", img.admin_user));
    env.push(format!("BIFROST_ADMIN_PASS={}", img.admin_password));

    if let Some(boltwall) = boltwall {
        if let Some(api_token) = &boltwall.stakwork_secret {
            env.push(format!("BIFROST_PROVISIONING_TOKEN={}", api_token));
        }
    }

    // …rest unchanged…
}
```

The call site in `images/mod.rs` (or wherever `bifrost()` is invoked)
needs to be updated to pass `boltwall`. This is identical to what
`repo2graph()` already does.

### B1.4 Expose admin creds via super-admin details

Hive consumes admin creds via the existing
`GET /super/details?id=<swarm_id>` endpoint that already returns
`x_api_key` (see `sphinx-swarm/src/bin/super/cmd.rs:256` for the
response struct).

Add two fields to the response payload:

```rust
pub struct SuperSwarmData {
    pub address: String,
    pub ec2_id: String,
    pub x_api_key: String,
    pub bifrost_admin_user: String,      // ← new
    pub bifrost_admin_password: String,  // ← new
    // …
}
```

Populate them from the swarm's persisted `BifrostImage` state in
`get_swarm_details_by_id` (`sphinx-swarm/src/bin/super/util.rs:1574`).

**Optional alternative**: don't include them in the existing details
payload (it's already large and surfaces in places we may not want to
leak credentials). Instead, add a separate
`GET /super/bifrost-credentials?id=<swarm_id>` endpoint, also
`x-super-token`-gated. Cleaner separation, one extra route. Either
works; the cost is roughly the same on both sides.

## Step B2 — hive changes

Three changes in Hive, in the order a workspace admin's journey hits them:

1. **B2.1** — server-side bootstrap: pull admin creds out of the
   gateway once and cache them in the DB (the existing
   `swarm.bifrostAdminUser` / `swarm.bifrostAdminPassword` columns).
2. **B2.2** — wire bootstrap into `resolveBifrost` so the existing
   reconciler picks creds up automatically.
3. **B2.3** — new HTTP endpoint that the frontend calls to display
   "visit your LLM dashboard" with the password.

The crucial invariant: **the `stakwork_secret` never leaves Hive's
Node.js process**. Only the per-gateway `admin_password` is ever
sent to a browser, and only to authenticated workspace admins.

### B2.1 Add a bootstrap function

```ts
// hive/src/services/bifrost/bootstrap.ts
export async function bootstrapAdminCreds(
  workspaceId: string,
): Promise<{ adminUser: string; adminPassword: string }> {
  const swarm = await db.swarm.findUniqueOrThrow({
    where: { workspaceId },
    select: { id: true, swarmId: true, swarmUrl: true /* … */ },
  });
  if (!swarm.swarmUrl) throw new BifrostConfigError(...);

  // Pull the stakwork shared secret from swarm super-admin. This is
  // the same value swarm passes to the gateway container as
  // BIFROST_PROVISIONING_TOKEN, so we can present it as a Bearer
  // token to /_plugin/admin-credentials.
  const stakworkSecret = await getStakworkSecretForSwarm(swarm.id);

  // /_plugin/* lives on the gateway's public port (8181 in dev,
  // whatever swarm exposes in prod), reached via TLS at the same
  // hostname as the rest of bifrost.
  const baseUrl = deriveBifrostBaseUrl(swarm.swarmUrl);

  const res = await fetch(`${baseUrl}/_plugin/admin-credentials`, {
    method: 'GET',
    headers: { Authorization: `Bearer ${stakworkSecret}` },
  });
  if (!res.ok) {
    throw new BifrostConfigError(
      `bootstrap failed: ${res.status} ${await res.text()}`,
    );
  }
  const body = (await res.json()) as {
    admin_username: string;
    admin_password: string;
  };

  // Persist encrypted, same idiom as swarmApiKey
  // (hive/src/services/swarm/db.ts:117).
  const enc = EncryptionService.getInstance();
  await db.swarm.update({
    where: { workspaceId },
    data: {
      bifrostAdminUser: body.admin_username,
      bifrostAdminPassword: JSON.stringify(
        enc.encryptField('bifrostAdminPassword', body.admin_password),
      ),
    },
  });

  return { adminUser: body.admin_username, adminPassword: body.admin_password };
}
```

### B2.2 Lazy invocation in `resolveBifrost`

Today (`hive/src/services/bifrost/resolve.ts:70`):

```ts
if (!swarm.bifrostAdminUser || !swarm.bifrostAdminPassword) {
  throw new BifrostConfigError(...);
}
```

Becomes:

```ts
if (!swarm.bifrostAdminUser || !swarm.bifrostAdminPassword) {
  // Lazy bootstrap. Idempotent — calling it twice on a healthy
  // swarm just reads + re-encrypts the same values.
  const { adminUser, adminPassword } = await bootstrapAdminCreds(workspaceId);
  return { baseUrl, adminUser, adminPassword };
}
```

This is the only consumer change. The existing reconciler
(`reconciler.ts`) and `BifrostClient` need no edits.

### B2.3 Frontend dashboard-credential endpoint

The frontend needs a way to render "visit your LLM dashboard" with
the password the workspace admin can copy. Add a new Next.js route:

```ts
// hive/src/app/api/workspaces/[slug]/bifrost/credentials/route.ts
//
// Returns the workspace's Bifrost dashboard URL + admin user +
// admin password (plaintext, decrypted) for display in the UI.
//
// Auth: workspace owner or admin role only — this is a privileged
// endpoint that returns plaintext credentials to the caller's
// browser.
//
// Idempotency: if creds are missing in the DB, the handler kicks off
// bootstrap (B2.1) before returning. So the very first call after
// swarm provisioning still gets a populated response.
export async function GET(
  request: NextRequest,
  { params }: { params: Promise<{ slug: string }> },
): Promise<NextResponse> {
  const session = await getServerSession(authOptions);
  if (!session?.user) return unauthorized();

  const { slug } = await params;
  const workspace = await db.workspace.findUnique({ where: { slug } });
  if (!workspace) return notFound();

  // Only workspace owners + admins can see the password. Plain
  // members get a 403 — they can use the gateway via Hive but not
  // log into the Bifrost dashboard directly.
  const role = await getWorkspaceRoleForUser(workspace.id, session.user.id);
  if (role !== 'OWNER' && role !== 'ADMIN') return forbidden();

  // resolveBifrost triggers bootstrap if creds are missing.
  // After this call, swarm.bifrostAdminUser / bifrostAdminPassword
  // are guaranteed populated.
  const { baseUrl, adminUser, adminPassword } = await resolveBifrost(workspace.id);

  return NextResponse.json({
    dashboardUrl: baseUrl,            // user clicks → bifrost UI login
    adminUser,                         // typically "admin"
    adminPassword,                     // plaintext, 16 chars
  });
}
```

Frontend usage (sketch):

```tsx
// SettingsPage component
const { data } = useQuery({
  queryKey: ['bifrost-credentials', workspace.slug],
  queryFn: () => fetch(`/api/workspaces/${workspace.slug}/bifrost/credentials`).then(r => r.json()),
});

return (
  <Card>
    <h3>LLM Gateway Dashboard</h3>
    <a href={data.dashboardUrl} target="_blank">Open dashboard</a>
    <div>Username: <code>{data.adminUser}</code></div>
    <div>
      Password: <code>{data.adminPassword}</code>
      <CopyButton text={data.adminPassword} />
    </div>
    <p className="text-sm">
      Sign in to the Bifrost dashboard to view real-time LLM usage,
      costs, and per-agent metrics.
    </p>
  </Card>
);
```

Why this endpoint instead of including the password in some existing
workspace payload:

- **Principle of least surprise**: a user staring at network traffic
  in DevTools should see admin credentials in a clearly-named
  endpoint, not buried in a generic workspace response.
- **Cache control**: this endpoint is `private, no-store` so
  passwords don't end up in CDN / proxy caches. A generic
  `/api/workspaces/<slug>` response might not be.
- **Audit logs**: easy to log "user X requested bifrost creds for
  workspace Y" at one place, without polluting the workspace-read
  audit stream.
- **Rotation UX**: this is also the obvious place to add `POST`
  for "rotate Bifrost admin password" once that's a feature
  (returns new creds, triggers gateway restart with new env).

### B2.4 Where does `stakworkSecret` come from?

In Hive today it's referenced from a few places (search
`stakworkService().createSecret` — see
`hive/src/app/api/swarm/route.ts:315`) and is the same shared key
swarm uses. Wire `getStakworkSecretForSwarm()` to whatever Hive
already does to access it for `createSecret`; the value doesn't need
to be fetched fresh, it can come from env or from the swarm's
super-admin details payload at swarm-creation time. (If it isn't
already persisted per-swarm, this is the right time to do so.)

**Whatever the source: it MUST be a server-side read.** No public
API endpoint, no client-side fetch, no inclusion in any payload sent
to a browser. The lookup function should only ever be called from
inside Next.js route handlers and background workers, never from
React components.

## Failure modes and idempotency

The bootstrap is designed to be re-runnable. Each component handles
the "what if I'm called twice" case explicitly:

| Scenario | Behaviour |
|---|---|
| Hive backend calls `bootstrapAdminCreds` twice in parallel (e.g. two browser tabs hitting `/bifrost/credentials` simultaneously) | Both return the same plaintext; second `db.swarm.update` overwrites with identical values. Mild waste, no data corruption. Worth adding a per-workspace mutex if this becomes hot, but it's a non-issue for v1. |
| `BIFROST_ADMIN_PASS` is rotated by swarm (new pw injected) | Bifrost re-hashes and flushes sessions on next boot. Hive's stored value is now wrong → all `/api/governance/*` calls 401 → next `resolveBifrost` should detect and re-bootstrap. (See open question below.) ALSO: the password shown in Hive's UI is now stale until re-bootstrap; a frontend that caches the response will be confusing — set short cache TTL or invalidate on a 401 from any subsequent governance call. |
| `BIFROST_PROVISIONING_TOKEN` doesn't match | Plugin returns 401, `bootstrapAdminCreds` throws `BifrostConfigError`. The `/bifrost/credentials` endpoint returns 500 to the frontend; UI should render "couldn't reach LLM dashboard — contact support" rather than a raw error. |
| Gateway is unreachable | Bootstrap throws on fetch (timeout). `resolveBifrost` rethrows. Frontend should render "LLM dashboard not yet available — try again in a minute" (the swarm may still be provisioning). The reconciler retries on its own tick. |
| Plugin env vars (`BIFROST_ADMIN_*`) not set | Plugin's `startPluginServer` logs a warning and the route doesn't register. `bootstrapAdminCreds` gets a 404 from the wrapper (wrapper returns 503 when the plugin proxy is disabled — see wrapper main.go). Surface that to the operator — this means swarm forgot to inject the env vars. |
| Workspace member (not admin/owner) hits `/bifrost/credentials` | 403 from the Hive endpoint. No bootstrap is triggered, no DB read happens. The check is the very first thing the handler does. |

### Open question: how does Hive detect a password rotation?

A clean detection signal would be: on a 401 from any
`/api/governance/*` call in `BifrostClient`, treat it as
"creds-may-be-stale", re-bootstrap, and retry once. That's a
~10-line change in `BifrostClient.request` (catch 401 → call
bootstrap → retry). Worth doing as soon as we expect password
rotation to be a thing (which is "never on the auto-rollout path but
yes when humans rotate them manually").

For now, document the manual recovery path:
> If the swarm rotates `BIFROST_ADMIN_PASS`, clear the affected
> `swarm.bifrostAdminUser` / `swarm.bifrostAdminPassword` rows in
> Hive's DB. The next `resolveBifrost` call re-bootstraps.

## What this doesn't cover

- **The gateway image itself**. Already done; see the gateway README.
- **Phase 1 reconciler.** Phase 1 assumes Bifrost admin creds are
  present. Phase 3 makes that assumption true.
- **Hot enforcement** (macaroon verification, budget enforcement,
  tool-loop heuristic). Those are phases 4+, still gated on
  `llm-governance-v2.md`.
- **Per-tenant Bifrost provisioning at scale**. We're standing up
  one gateway per swarm. Multi-tenant single-gateway is a separate
  problem (see `agent-registry.md`).

## Rollout order

1. **Publish the gateway image.** Versioned `stakgraph-gateway:<tag>`
   from this repo (one-time CI step; the Makefile already has
   `docker-build`).
2. **Land B1 in sphinx-swarm.** Image switch + admin-credential
   fields + env injection + super-admin details payload. Wait for it
   to ship to swarm fleet.
3. **Land B2.1 + B2.2 in Hive (server-only).** Bootstrap function
   + lazy invocation in `resolveBifrost`. The phase-1 reconciler
   starts working against new swarms automatically.
4. **Land B2.3 in Hive (frontend exposure).** The
   `/api/workspaces/[slug]/bifrost/credentials` endpoint, plus
   whatever UI surface renders the "open dashboard / copy password"
   card. Can ship behind a feature flag — when disabled, the
   reconciler still works, the dashboard is just not linked in the
   UI.
5. **Backfill existing swarms.** For swarms that pre-date the image
   switch, force-re-pull the gateway image and restart. Hive
   bootstraps lazily on the next `resolveBifrost` call (which fires
   from either the reconciler tick or the frontend hitting
   `/bifrost/credentials`).

If you need to ship 3 and 4 together (no dual deploy), gate the
backend on the same flag and release as one PR — same effect.
